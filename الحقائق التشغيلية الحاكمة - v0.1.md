## حقائق حاكمة

1. الواجهة الرئيسية المعتمدة للمستخدم هي `Open WebUI` على `http://localhost:3000`.
2. خدمة `open-terminal` تعمل على `http://localhost:8000`، وليست لوحة التحكم الرئيسية.
3. المجلد المحلي `saleh/` يظهر داخل `open-terminal` على المسار `/home/user/projects`.
4. خدمات التشغيل الحالية ضمن `docker-compose.yml` تشمل: `open_webui`, `postgres`, `chromadb`, `pipelines`, `tika`, `searxng`, `n8n`, `knowledge_watcher`, `data_pipeline`.
   خدمات تحت profile (لا تبدأ تلقائياً): `open-terminal`, `browserless`, `crawl4ai`, `python_agent`.
5. خدمة `mcpo` ملغاة في هذا المشروع حاليًا، ولا تُستخدم كمسار تشغيلي.
6. أي مواد تاريخية مرتبطة بـ `mcpo` لا تُعامل كمرجع تشغيلي حي.
7. لا توجد خدمة `Code Server` مفعلة في `docker-compose.yml` الحالي.
8. لا توجد خدمة `AnythingLLM` مفعلة في التشغيل الحالي.
9. خدمة استيعاب الملفات البشرية ليست واجهة مستقلة؛ المنفذ `8001` مخصص لـ `data_pipeline`.
10. مسار استيعاب المعرفة الحالي هو: `knowledge_inbox/` → `knowledge_watcher` → `data_pipeline` → `ChromaDB`.
11. الملفات الناجحة من الاستيعاب تنتقل إلى `knowledge_processed/`.
12. الملفات الفاشلة من الاستيعاب تنتقل إلى `knowledge_failed/`.
13. خدمة `knowledge_watcher` الحية الحالية تُبنى من `file_watcher/`، والملف التنفيذي المعتمد هو `file_watcher/watcher.py`.
14. المجلد `knowledge_processing/` ليس وجهة نجاح حية في التشغيل الحالي.
15. المجلد `knowledge_archive/` ليس وجهة نجاح لـ watcher الحالي، لكنه يبقى مستخدمًا لأرشفة بعض المصادر مثل `knowledge_archive/uqn/`.
16. المجلد `saleh_dashboard/` ليس خدمة مفعلة في `docker-compose.yml` الحالي، ولا يمثل الواجهة الحية المعتمدة للمستخدم.
17. ChromaDB المستخدمة فعليًا في المسارات الحية تعمل عبر واجهات `v1` وليس `v2`.
18. ~مجموعة المعرفة الحية المعتمدة حالياً هي `saleh_knowledge`~ (محدث — انظر #32). مجموعة المعرفة الحية المعتمدة هي `saleh_knowledge_qwen3`.
19. ~نموذج التضمين الحي المعتمد هو `nomic-embed-text:latest`~ (محدث — انظر #32). نموذج التضمين الحي المعتمد هو `qwen3-embedding:0.6b` (1024 dims).
20. لا يجوز لأي وكيل افتراض توفر endpoint أو خدمة أو مجموعة بيانات بناءً على وثيقة قديمة دون مطابقته مع `docker-compose.yml` أو هذه الوثيقة.
21. سير عمل n8n "Daily Backup" كان قبل 2026-04-06 يفحص فقط ChromaDB عبر HTTP GET ولم يكن يستدعي `backup.ps1` إطلاقاً — كل تقارير "نجاح النسخ الاحتياطي" قبل هذا التاريخ كانت مضللة.
22. "نجاح تنفيذ n8n" يعني فقط أن عُقد سير العمل أكملت تشغيلها — لا يعني إنشاء ملفات أو نجاح عمليات خارجية. التحقق الفعلي يتطلب فحص stdout من executeCommand.
23. الذكاء الاصطناعي (الرئيس/النموذج) لا يستطيع تعديل ملفات النظام (.py، .ps1، .json) من المحادثة — فقط يمكنه تصميم JSON لسير عمل جديدة ثم يستوردها المستخدم يدوياً.
24. نوع العقدة `n8n-nodes-base.executeCommand` غير مدرج في القائمة المسموحة للإنشاء التلقائي عبر واجهة المحادثة — لا يمكن للذكاء الاصطناعي إنشاء سير عمل تنفّذ أوامر النظام تلقائياً.
25. النسخ الاحتياطية التي تم التحقق منها فعلياً: `backups/20260327_222639.zip`، `backups/20260330_005300.zip`، `backups/auto_backups/` (آخرها 2026-04-03). لا توجد نسخة احتياطية موثّقة بعد 2026-04-03.
26. خدمة `python_agent` معطلة اعتباراً من 2026-04-06 وموضوعة تحت `profiles: python-agent`. لا تُفعَّل إلا عند تحديد وظيفة حقيقية وموثقة لها.
27. أي حاوية تمتلك `/var/run/docker.sock` يجب أن يكون لها وظيفة صريحة ومعروفة وموثقة. الامتلاك بدون وظيفة = خطر أمني.
28. أي كود توليده نموذج AI ويدّعي قدرات تلقائية (مراقبة، scraping، حقن) يُعامل كمسودة تجريبية — لا يُنفَّذ قبل مراجعة بشرية. مجلد الأرشيف للكود الخطر: `archive/cleanup_20260406/president_scripts/`.
29. المنهج الحاكم لقرارات التطوير: لا نريد المستحيل، نريد الأفضل الممكن فقط. أي ميزة لا تتصل بالنظام الحي أو لا يمكن التحقق منها تُعامل كخيال حتى يثبت العكس.
30. تنفيذ كود Python في المحادثات يتم عبر `pyodide_code_execution` (Tool مثبَّت في Open WebUI) — يشغّل Python في WebAssembly داخل المتصفح، معزول كاملاً عن الـ server. `tools/code_interpreter.py` أُرشف 2026-04-06 لأنه كان يشغّل Python على OS الـ host مباشرة (subprocess) وهو أضعف أمنياً.
31. أدوات Open WebUI المثبَّتة: 17 Tool نشطة بما فيها pyodide_code_execution، web_search_and_crawl، arxiv_search_tool، comfyui_image_video_generator. القائمة الكاملة عبر: GET /api/v1/tools/
32. مجموعة ChromaDB الوحيدة السارية هي `saleh_knowledge_qwen3` (11,566 chunk، 1024 dims، cosine distance). مجموعتا `saleh_knowledge` (2 وثيقة خاطئة) و `ceo_knowledge_base` (0 docs) حُذفتا نهائياً 2026-04-06.
33. نموذج التضمين المعتمد هو `qwen3-embedding:0.6b` (1024 dims). لا يجوز استخدام `nomic-embed-text` (768 dims) مع هذه المجموعة للتضمين، سيخفق بسبب تعارض الأبعاد.
34. Ollama يعمل على الجهاز المضيف (host) لا داخل Docker. في داخل الحاويات يُصاح عبر `host.docker.internal:11434`. مسجَّل في Windows Registry `HKCU:\Software\Microsoft\Windows\CurrentVersion\Run\Ollama` للتشغيل التلقائي عند إقلاع الجهاز.
35. Docker Desktop مضبوط على `AutoStart=true` في `%APPDATA%\Docker\settings-store.json`. جميع الحاويات تستأنف تلقائياً بسبب `restart: unless-stopped`. تسلسل الإقلاع: Docker Desktop → Ollama → جميع الحاويات (تلقائي).
36. سجل المتابعة والتحسين المستمر المعتمد هو `logs/ops_journal.jsonl` بصيغة JSONL. كل إجراء تشغيلي مؤثر (backup، restart، incident fix، تغيير إعدادات) يجب أن يدوَّن فيه مع: الحالة (`ok|warn|fail`) + ملخص + الخطوة التالية.
37. تم التحقق عملياً من مسار RAG end-to-end بتاريخ 2026-04-06 عبر الاستعلام: "ما هي شروط الفسخ في العقود القانونية السعودية؟". النتيجة: embedding (1024 dims) + retrieval (`15` chunks من `saleh_knowledge_qwen3`) + context injection + LLM answer all passed. المرجع التشغيلي: `logs/ops_journal.jsonl` (action=`rag_e2e_verified`).
